# Technical Risk Register

Per [`MASTER_SPEC.md`](./MASTER_SPEC.md) §37. Risks are rated by **Impact** (L/M/H) and **Likelihood** (L/M/H). Mitigations reference the owning design doc.

---

| #   | Risk                                                                           | Impact | Likelihood | Mitigation                                                                                                                                                                    |
| --- | ------------------------------------------------------------------------------ | :----: | :--------: | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R1  | **Cross-tenant data leak** (RLS gap or trusting client `tenant_id`)            |   H    |     M      | Default-deny RLS on every tenant table; server re-checks tenant+permission; automated RLS tests per phase; never trust client tenant_id. [`SECURITY.md`](./SECURITY.md)       |
| R2  | **Secret exposure to browser** (service-role/provider keys)                    |   H    |     M      | Secrets server-side/Edge only; only metadata in DB; secret scanning in CI; env validation. [`SECURITY.md`](./SECURITY.md) §4                                                  |
| R3  | **AI hallucinates a project fact** (price/availability/offer)                  |   H    |     M      | Hard grounding to Approved sources; no-source → escalate; inventory-freshness gating; never recommend unavailable units. [`AI_SYSTEM.md`](./AI_SYSTEM.md) §4                  |
| R4  | **Prompt injection via uploaded docs/website text**                            |   H    |     M      | Treat retrieved content as untrusted reference, never instructions; delimited context; output validation. [`SECURITY.md`](./SECURITY.md) §9                                   |
| R5  | **Unsafe duplicate merge / data loss**                                         |   H    |     L      | Merge preserves all children, is reversible, audited; auto-merge only exact + opt-in; review queue otherwise. [`CONTRADICTIONS.md`](./CONTRADICTIONS.md) C10                  |
| R6  | **Lost/duplicated inbound leads** (webhook retries, failures)                  |   H    |     M      | Signature verification, idempotency keys, raw-event store, durable queue + DLQ + replay. [`INTEGRATIONS.md`](./INTEGRATIONS.md) §1                                            |
| R7  | **WhatsApp policy/template violations** (messaging window, templates, opt-out) |   H    |     M      | Respect 24h session + approved templates; DNC/consent enforced pre-send; reactive vs proactive split. [`CONTRADICTIONS.md`](./CONTRADICTIONS.md) C4                           |
| R8  | **AI cost overrun**                                                            |   M    |     M      | Cheap-model routing for high-volume tasks; per-tenant budgets with soft/hard limits; full usage tracking. [`AI_SYSTEM.md`](./AI_SYSTEM.md) §2.3                               |
| R9  | **Scoring not reproducible / disputed**                                        |   M    |     M      | Deterministic engine, rule-version stamping, full history, simulation; AI never writes the score. [`SCORING_ENGINE.md`](./SCORING_ENGINE.md)                                  |
| R10 | **Provider outage / breaking API change** (AI or messaging)                    |   M    |     M      | Adapter abstraction + fallback order; health checks; queue retries with backoff. [`AI_SYSTEM.md`](./AI_SYSTEM.md), [`INTEGRATIONS.md`](./INTEGRATIONS.md)                     |
| R11 | **Background jobs run inline / lost on navigation**                            |   M    |     M      | Mandatory durable jobs for multi-step/external work; fast-ack endpoints. [`ARCHITECTURE.md`](./ARCHITECTURE.md) §4                                                            |
| R12 | **Mobile experience degraded** (shrunk tables)                                 |   M    |     M      | Purpose-built mobile layouts, bottom nav, sticky actions; accessibility + touch-target tests. [`UI_SYSTEM.md`](./UI_SYSTEM.md) §5                                             |
| R13 | **PII / compliance exposure** (logs, exports, retention)                       |   H    |     L      | PII masking by permission; export logging; retention/deletion workflows; log redaction. [`SECURITY.md`](./SECURITY.md) §10                                                    |
| R14 | **Portal ingestion fragility** (no stable APIs)                                |   M    |     H      | Multiple fallbacks: Gmail parsing, CSV/XLSX, generic webhook, mapping wizard; per-row error reporting. [`INTEGRATIONS.md`](./INTEGRATIONS.md) §2                              |
| R15 | **Multilingual accuracy** (6 languages; preserving names/amounts)              |   M    |     M      | Language detection per message; preserve project names/currency/units verbatim; quality reviews. [`AI_SYSTEM.md`](./AI_SYSTEM.md) §4.1                                        |
| R16 | **Predictive scoring bias / misuse**                                           |   H    |     L      | Off by default; never protected attributes; never cross-tenant; validated + approved before influence; shown beside rule score. [`SCORING_ENGINE.md`](./SCORING_ENGINE.md) §6 |
| R17 | **Vector/FTS retrieval quality drift** as corpus grows                         |   M    |     M      | Hybrid retrieval, re-indexing, AI-answer tester, evaluation cases. [`AI_SYSTEM.md`](./AI_SYSTEM.md) §3                                                                        |
| R18 | **Scope/velocity risk** (very large spec)                                      |   M    |     H      | Strict phasing; per-phase exit gates; BUILD_STATUS tracking; no uncontrolled single pass. [`IMPLEMENTATION_PLAN.md`](./IMPLEMENTATION_PLAN.md)                                |
| R19 | **Dedicated vs shared mode divergence**                                        |   M    |     L      | Single code path; deployment mode is env config only; no per-customer forks. [`ARCHITECTURE.md`](./ARCHITECTURE.md) §3.4                                                      |
| R20 | **Migration mistakes in production**                                           |   H    |     L      | Forward-only reviewed migrations; staging validation; backups + restore drills; destructive change = stop-condition. [`DEPLOYMENT.md`](./DEPLOYMENT.md)                       |

## Top risks to watch first

R1, R2, R3, R4, R6 are the highest combined exposure and are addressed structurally in Phases 1, 3, 4 and 5 — RLS and secret handling land in Phase 1 before any tenant data exists; grounding and injection defenses land with the AI phase before any buyer-facing message is sent.
